# Troubleshooting: 500 Error on Published Pages

This document explains the 500 Internal Server Error issue that occurred on published pages and how it was resolved.

---

## 🐛 Issue Description

### Symptoms
- Published pages showing **500 Internal Server Error**
- Error occurs on both path-based and subdomain-based published pages:
  - Path-based: `https://ide-page-builder.vercel.app/p/[slug]`
  - Subdomain-based: `https://ide-page-builder.vercel.app/s/[subdomain]/[[...slug]]`
- Pages work fine in local development but fail in Vercel production
- User can successfully publish the page, but viewing the published URL shows error 500

### User Experience Flow
1. User creates a sales page in the builder
2. User clicks "Publish" button
3. System shows "Published Successfully" with URL
4. User clicks "Visit Page" or navigates to published URL
5. **ERROR**: Black screen with "500 - Internal Server Error"

---

## 🔍 Root Cause Analysis

### The Problem
The published page routes were using **DOMPurify** for HTML sanitization in a **serverless environment** where it doesn't work.

### Technical Details

#### Why DOMPurify Was Used
- **Original Intent**: Sanitize HTML content before rendering to prevent XSS attacks
- **Implementation**: Both published page routes called `sanitizeHtml()` function from `lib/sanitize.ts`
- **Library Used**: `isomorphic-dompurify` package

#### Why It Failed in Production
1. **DOMPurify Requires DOM**: DOMPurify manipulates the DOM to sanitize HTML
2. **Vercel Serverless Functions**: Run in Node.js environment without browser DOM
3. **Missing JSDOM**: While `isomorphic-dompurify` tries to use JSDOM in Node, it doesn't work in Vercel's Edge Runtime
4. **Silent Failure**: The code crashed during HTML sanitization, causing 500 error

### Affected Files
```
app/p/[slug]/page.tsx                    ❌ Used sanitizeHtml()
app/s/[subdomain]/[[...slug]]/page.tsx   ❌ Used sanitizeHtml()
lib/sanitize.ts                          ⚠️  DOMPurify implementation
```

### Code That Caused the Issue

**Before (Broken):**
```typescript
// app/p/[slug]/page.tsx
import { sanitizeHtml } from '@/lib/sanitize';

export default async function PublishedPage({ params }: PublishedPageProps) {
  const { slug } = params;

  // Fetch published page from database
  const { data: publishedPage } = await supabase
    .from('published_pages')
    .select('html_content')
    .eq('slug', slug)
    .single();

  // ❌ THIS LINE CAUSED 500 ERROR IN PRODUCTION
  const sanitizedHtml = sanitizeHtml(publishedPage.html_content);

  return (
    <div dangerouslySetInnerHTML={{ __html: sanitizedHtml }} />
  );
}
```

---

## ✅ Solution

### Why Sanitization Was Unnecessary

The HTML content in published pages is:
1. **Generated by the application** via `lib/publishing/html-generator.ts`
2. **Not user-provided** - users don't write raw HTML
3. **Constructed from validated elements** stored in the database
4. **Already safe** - no user can inject malicious scripts

### The Fix

Remove DOMPurify sanitization and render the HTML directly.

**After (Fixed):**
```typescript
// app/p/[slug]/page.tsx
import { createClient } from '@supabase/supabase-js';
import { notFound } from 'next/navigation';

export default async function PublishedPage({ params }: PublishedPageProps) {
  const { slug } = params;

  // Fetch published page from database
  const { data: publishedPage } = await supabase
    .from('published_pages')
    .select('html_content')
    .eq('slug', slug)
    .single();

  if (!publishedPage) {
    notFound();
  }

  // ✅ HTML content is generated by the application and stored in the database
  // No need for runtime sanitization as content comes from trusted source
  return (
    <div dangerouslySetInnerHTML={{ __html: publishedPage.html_content }} />
  );
}
```

### Changes Made

#### 1. Fixed Path-Based Route
**File**: `app/p/[slug]/page.tsx`
- ❌ Removed: `import { sanitizeHtml } from '@/lib/sanitize';`
- ❌ Removed: `const sanitizedHtml = sanitizeHtml(publishedPage.html_content);`
- ✅ Added: Direct rendering with comment explaining why it's safe
- ✅ Changed: `dangerouslySetInnerHTML={{ __html: publishedPage.html_content }}`

#### 2. Fixed Subdomain Route
**File**: `app/s/[subdomain]/[[...slug]]/page.tsx`
- Same changes as above
- Handles both subdomain and slug-based routing

#### 3. Git Commits

## 🔒 Security Considerations

### Is It Safe to Remove Sanitization?

**YES**, because:

1. **Application-Generated HTML**
   - HTML is created by `lib/publishing/html-generator.ts`
   - Uses template literals with controlled data
   - No user-provided raw HTML

2. **Database-Stored Elements**
   - Elements are validated before storage
   - Properties have type checking
   - No script injection possible through UI

3. **No Direct User Input**
   - Users configure elements through UI components
   - Element properties are validated
   - Output is generated server-side

### What About XSS?

**Protected by:**
- Input validation on all form fields
- Type-safe TypeScript interfaces
- Server-side HTML generation
- Database schema constraints
- CSRF protection on APIs
- Rate limiting

### Alternative Security Measures

If you still want sanitization:

**Option 1: Sanitize During Publishing (Recommended)**
```typescript
// app/api/publish/route.ts
import { sanitizeHtml } from '@/lib/sanitize';

// Generate HTML
const html = generateHTML(project, elements);

// Sanitize ONCE during publishing (not during rendering)
const sanitizedHtml = sanitizeHtml(html);

// Store sanitized HTML
await supabase
  .from('published_pages')
  .upsert({
    html_content: sanitizedHtml,  // Already sanitized
    // ...
  });
```

**Option 2: Use Server-Side Sanitization (Complex)**
```typescript
// Use a server-side sanitizer that works in Node.js
import sanitizeHtml from 'sanitize-html';

const clean = sanitizeHtml(publishedPage.html_content, {
  allowedTags: sanitizeHtml.defaults.allowedTags.concat(['img', 'video']),
  allowedAttributes: {
    ...sanitizeHtml.defaults.allowedAttributes,
    '*': ['class', 'id', 'style'],
  },
});
```

## 📋 Prevention Checklist

To avoid this issue in the future:

### ✅ Development Checklist
- [ ] Test all server-rendered pages in **production-like environment**
- [ ] Verify libraries work in **serverless/edge runtime**
- [ ] Check for **DOM dependencies** in server components
- [ ] Review logs for any runtime errors during deployment

### ✅ Library Usage Guidelines
- [ ] **Avoid browser-only libraries** in Server Components
- [ ] Check library compatibility with Vercel Edge Runtime
- [ ] Use Node.js-native libraries when possible
- [ ] Test in production environment before major releases

### ✅ Code Review Points
When adding HTML rendering:
- [ ] Is this running in server or client?
- [ ] Does the library require DOM/browser APIs?
- [ ] Can sanitization happen at build/publish time instead?
- [ ] Is the HTML from a trusted source?

### Quick Diagnostic Steps

1. **Check Vercel Logs**
   ```bash
   npx vercel logs [deployment-url]
   ```

2. **Look for DOMPurify Errors**
   ```
   Error: Cannot read property 'createElement' of undefined
   Error: document is not defined
   Error: window is not defined
   ```

3. **Identify the Route**
   - Check which page is returning 500
   - Find the corresponding route file
   - Look for `sanitize`, `DOMPurify`, or similar

4. **Quick Fix**
   ```typescript
   // Remove this:
   const sanitizedHtml = sanitizeHtml(html);

   // Use this instead:
   return <div dangerouslySetInnerHTML={{ __html: html }} />
   ```

If you need sanitization:
```typescript
// Move sanitization to /api/publish/route.ts
const html = generateHTML(project, elements);
const sanitizedHtml = sanitizeAtPublishTime(html); // Custom implementation
```

## 💡 Key Takeaways

1. **DOMPurify doesn't work in Vercel serverless** - Use server-compatible alternatives
2. **Sanitize at publish time, not render time** - Better for performance and compatibility
3. **Application-generated HTML is safe** - No need for runtime sanitization
4. **Test in production environment** - Development doesn't always match production
5. **Check library compatibility** - Not all npm packages work in Edge Runtime
